# Migrate Artifact CRUD to Supabase Client (SOLO-98) - Design Document

## Goal Description

The goal of this task is to migrate the Next.js web application away from mock API endpoints (`/api/artifacts/*`) to direct interactions with the Supabase database. We will adopt modern React and Next.js paradigms by utilizing **React Server Components (RSCs)** for data fetching and **Server Actions** for mutations.

## Proposed Architecture

### 1. Data Fetching (Server Components)
We will convert the following pages from Client Components to Server Components:
*   `app/artifacts/page.tsx`
*   `app/study/page.tsx` (If interactions require client state like flashcards flipping, we may keep a client shell but pass initial data as props from the Server Page, or lift data fetching to a Server Component wrapper layout/page).

The fetching logic will bypass the `/api/artifacts` network requests entirely and query Supabase directly using `@supabase/ssr` server utilities. 

### 2. Mutations (Server Actions)
We will create a new file `app/actions/artifacts.ts` (or similar) to handle data mutations, completely removing the need for `POST`, `PUT`, and `DELETE` handlers inside `app/api/artifacts/route.ts` and `[id]/route.ts`.
This will include:
*   `deleteArtifact(id: string)`

These Server Actions will be invoked directly from Client Components (e.g., delete buttons).

### 3. Client & Server Supabase Utilities
We need to set up the standard `@supabase/ssr` files:
*   `lib/supabase/server.ts` (for Server Components, Actions, Route Handlers)
*   `lib/supabase/client.ts` (for Client Components, if needed)

We will also install the necessary packages:
`npm install @supabase/supabase-js @supabase/ssr` (or `pnpm`/`bun`/`yarn` equivalent for the project).

### 4. Cleanup
*   Remove the mock data (`MOCK_ARTIFACTS`, `ALL_PAIRS`) from the frontend files.
*   Remove `app/api/artifacts/route.ts` and `app/api/artifacts/[id]/route.ts` as their responsibilities will be fully replaced by RSCs and Server Actions.
